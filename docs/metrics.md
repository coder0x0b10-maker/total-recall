# Metrics Framework

Centralized metrics collection, quality evaluation, and reporting for the Total Recall pipeline.

## Overview

The metrics framework provides three capabilities:

1. **Pipeline Performance** — automatic timing of every agent stage (observer, reflector, sensor sweep, rumination, etc.)
2. **Memory Quality** — snapshot, diff, and score your observations.md
3. **Recall Accuracy** — fixture-based tests to verify the recovery pipeline works correctly

All metrics are written as **JSONL** to `memory/metrics/YYYY-MM-DD.jsonl`, making them queryable with `jq`, `grep`, and any tool that understands line-delimited JSON.

## Metric Schema

Each line in the JSONL file is one metric record:

```json
{
  "ts": "2026-04-03T10:00:00Z",
  "component": "observer",
  "metric": "latency_ms",
  "value": 3200,
  "extra": {
    "stage": "llm_call",
    "model": "qwen/qwen3.6-plus:free",
    "attempt": 1,
    "status": "ok"
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `ts` | string | UTC ISO timestamp |
| `component` | string | Agent name: `observer`, `reflector`, `sensor_sweep`, `rumination`, `preconscious` |
| `metric` | string | Metric name (see "Collected Metrics" below) |
| `value` | number | Numeric value |
| `extra` | object | Context fields (varies by metric) |

## Collected Metrics by Component

### Observer (`scripts/observer-agent.sh`)

| Metric | Value | Extra fields | Description |
|--------|-------|-------------|-------------|
| `transcripts_found` | count | — | JSONL files discovered in session dir |
| `lines_extracted` | count | — | Meaningful lines extracted from transcripts |
| `latency_ms` | duration | `stage`, `model`, `attempt`, `status` | Per-stage: transcript discovery, message extraction, LLM call, dedup, reflector trigger |
| `llm_latency_ms` | duration | `model`, `attempt`, `response_size`, `status` | Individual LLM API attempt timing |
| `reflector_run` | 0/1 | `status` | Whether reflector was triggered and its result |
| `total_words` | count | — | Words in observations.md after append |
| `run_complete` | 1 | `status` | End-of-run summary (1 = completed successfully) |

### Reflector (`scripts/reflector-agent.sh`)

| Metric | Value | Extra fields | Description |
|--------|-------|-------------|-------------|
| `latency_ms` | duration | `stage`, `model`, `attempt`, `response_size`, `status` | LLM attempt timing |
| `words_before` | count | — | Word count before consolidation |
| `words_after` | count | — | Word count after consolidation |
| `reduction_pct` | percent | — | Compression ratio: `(before - after) / before * 100` |
| `run_complete` | 1 | `status`, `words_before`, `words_after`, `reduction_pct` | End-of-run summary |

### Sensor Sweep (`scripts/sensor-sweep.sh`)

| Metric | Value | Extra fields | Description |
|--------|-------|-------------|-------------|
| `connector_duration_ms` | duration | `connector`, `exit_code` | Per-connector execution time |
| `events_emitted` | count | — | New events written to bus.jsonl |
| `events_pruned` | count | — | Old consumed events removed |
| `run_complete` | 1 | `events`, `pruned` | End-of-run summary |

### Rumination Engine (`scripts/rumination-engine.sh`)

| Metric | Value | Extra fields | Description |
|--------|-------|-------------|-------------|
| `latency_ms` | duration | `stage`, `model`, `tokens`, `status` | Per-stage: LLM rumination, classification, enrichment, lookup execution |
| `events_processed` | count | — | Events read from bus |
| `tokens_used` | count | — | LLM tokens consumed |
| `rumination_notes` | count | — | Notes generated by the model |
| `run_complete` | 1 | `events`, `tokens`, `notes`, `trigger` | End-of-run summary |

### Preconscious (`scripts/preconscious-select.sh`)

| Metric | Value | Extra fields | Description |
|--------|-------|-------------|-------------|
| `notes_collected` | count | — | Notes gathered from last 7 days |
| `buffer_word_count` | count | — | Final buffer size |
| `run_complete` | 1 | `notes`, `words` | End-of-run summary |

## How to Use

### Generate a Daily Report

```bash
# Report for today
bash scripts/metrics-report.sh

# Report for the last 7 days
bash scripts/metrics-report.sh --week

# Report for the last N days
bash scripts/metrics-report.sh --days 14
```

Output: `memory/metrics/daily-report-YYYY-MM-DD.md`

The report includes:
- Pipeline summary table (components, runs, avg/min/max latency)
- Observer and reflector detail
- Memory quality score
- Sensor sweep stats
- Dream cycle status
- Error summary with timestamps

### Evaluate Memory Quality

```bash
# Score current observations.md (0-100)
bash scripts/metrics-quality.sh score

# Take a snapshot of current state
bash scripts/metrics-quality.sh snapshot

# Compare two snapshots
bash scripts/metrics-quality.sh diff

# Compare two specific snapshots
bash scripts/metrics-quality.sh diff memory/metrics/snapshots/2026-04-03-1000.md memory/metrics/snapshots/2026-04-04-1000.md
```

The quality score (0-100) is based on six factors:

| Factor | Weight | What it measures |
|--------|--------|-----------------|
| Type metadata | 25 | Ratio of observations with `dc:type` tags |
| Type diversity | 20 | Number of different observation types present |
| Importance balance | 20 | Whether you have a mix of high/mid/low importance items |
| Size appropriate | 15 | Word count vs ideal range (1000-5000) |
| Content quality | 20 | Ratio of non-trivial observations (>15 chars) |
| Structure | 20 | Header, separators, observations present, date sections |

Grades: A (85+), B (70+), C (50+), D (30+), F (<30)

### Run Recall Accuracy Tests

```bash
# Run all tests
bash tests/metrics-recall-test.sh

# Run tests without cleaning up temp files (for debugging)
bash tests/metrics-recall-test.sh --no-cleanup
```

Test results are TAP-formatted to stdout and logged to `memory/metrics/recall-tests-YYYY-MM-DD.jsonl`.

Test cases:
1. **empty_input_handles_gracefully** — observer exits cleanly with no transcripts
2. **dedup_no_duplicates_on_rerun** — running observer twice produces <15% file growth
3. **flush_mode_executes** — flush mode doesn't crash
4. **large_session_150_plus_messages** — observer handles 200-line session files
5. **recovery_mode_detects_missed_session** — recovery mode recognizes the session file
6. **metrics_infrastructure_works** — metrics JSONL is written and valid

### Query Metrics Directly

Since metrics are JSONL, you can query them with standard tools:

```bash
# All LLM attempts that failed today
jq 'select(.metric == "latency_ms" and .extra.status == "failed")' memory/metrics/$(date +%Y-%m-%d).jsonl

# Average observer LLM latency
jq -r 'select(.metric == "latency_ms" and .component == "observer") | .value' memory/metrics/*.jsonl | awk '{s+=$1; n++} END {print s/n "ms"}'

# Count reflector runs this week
jq 'select(.metric == "run_complete" and .component == "reflector")' memory/metrics/*.jsonl | wc -l

# Per-connector performance
jq 'select(.metric == "connector_duration_ms") | "\(.extra.connector): \(.value)ms"' memory/metrics/$(date +%Y-%m-%d).jsonl
```

## Instrumenting New Scripts

To add metrics to another script, source the library and use the provided functions:

```bash
source "$SCRIPT_DIR/_metrics.sh"
metrics_init "$WORKSPACE"

# Wrap any code block with timing
stage_start "my_operation"
# ... your code ...
stage_end "my_operation" '{"context":"additional"}'

# Record arbitrary metrics
metrics_record "items_processed" 42 '{"type":"email"}'

# End-of-run summary
metrics_flush "my_component" '{"status":"ok","items":42}'
```

The component name is automatically derived from the sourcing script's filename.

## File Layout

```
scripts/
  _metrics.sh              # Core metrics library (sourced by agents)
  metrics-quality.sh       # Memory quality evaluator (snapshot/diff/score)
  metrics-report.sh        # Daily report generator

tests/
  metrics-recall-test.sh   # Recall accuracy test harness
  fixture-basic-session.jsonl  # Test fixture

memory/
  metrics/
    YYYY-MM-DD.jsonl       # Daily pipeline metrics
    recall-tests-YYYY-MM-DD.jsonl  # Test results
    daily-report-YYYY-MM-DD.md     # Generated reports
    quality/
      YYYY-MM-DD.json      # Quality score results
      snapshot-YYYY-MM-DD-HHMM.json  # Snapshot metadata
    snapshots/
      YYYY-MM-DD-HHMM.md   # observations.md snapshots
```
